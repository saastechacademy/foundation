# Receive Transfer Order Items

## Purpose

The current receiving flow uses a single API:

- Method: `POST`
- API: `{baseUrl}/rest/s1/poorti/transferOrders/{orderId}/receipts`

This API handles:
- receiving existing TO items
- receiving and closing TO items
- receiving a new product that is not on the original TO

## Behavior

1. The API receives quantity at the Transfer Order item level, not at the shipment level.
2. If shipped quantity for an item is spread across multiple shipments, OMS internally splits the incoming receipt across eligible shipment items in shipment order.
3. If received quantity is greater than the quantity available to receive from shipped transfer shipments, the remaining quantity is received directly against the TO item without shipment references.
4. A `ShipmentReceipt` record is created for each receipt generated by the service.
5. Inventory is received through `co.hotwax.poorti.FulfillmentServices.receive#ProductInventory`.
6. If `statusId` is sent for an existing TO item, the service updates the order item status before checking whether the TO can be completed.
7. If the Transfer Order is already in `ORDER_COMPLETED`, no further receipts are allowed.

## API Parameters

### Path parameter

1. `orderId` - The ID of the Transfer Order in OMS.

### Request body

1. `facilityId` - The facility in OMS where the Transfer Order is being received.
2. `receivedDateTime` - Optional timestamp for the receiving session. If not sent, OMS uses the current timestamp.
3. `items` - List of items to receive.

Each item supports:
1. `orderItemSeqId` - Transfer Order item sequence ID. Optional for new product receipts.
2. `productId` - Product being received.
3. `quantityAccepted` - Quantity to receive. Optional when the request is only meant to close the TO item.
4. `statusId` - Optional item status to apply. Use `ITEM_COMPLETED` for receive-and-close or close-only.

## Scenarios

### 1. Receive TO Items

Use this when the receiver wants to record received quantity and keep the item open unless OMS completes it based on the resulting status.

```json
{
  "facilityId": "BROADWAY",
  "items": [
    {
      "orderItemSeqId": "01",
      "productId": "10784",
      "quantityAccepted": 5
    },
    {
      "orderItemSeqId": "02",
      "productId": "10326",
      "quantityAccepted": 10
    }
  ]
}
```

### 2. Receive And Close TO Item

Use the same API and send `statusId: "ITEM_COMPLETED"` on the item.

#### Receive quantity and close

```json
{
  "facilityId": "WEST_JORDAN",
  "items": [
    {
      "orderItemSeqId": "01",
      "productId": "10079",
      "quantityAccepted": 1,
      "statusId": "ITEM_COMPLETED"
    }
  ]
}
```

#### Close without receiving additional quantity

```json
{
  "facilityId": "WEST_JORDAN",
  "items": [
    {
      "orderItemSeqId": "01",
      "productId": "10079",
      "statusId": "ITEM_COMPLETED"
    }
  ]
}
```

### 3. Receive New Item

If a product arrives that is not part of the original TO, send only `productId` and `quantityAccepted`. No `orderItemSeqId` is required.

```json
{
  "facilityId": "WEST_JORDAN",
  "items": [
    {
      "productId": "10079",
      "quantityAccepted": 1
    }
  ]
}
```

## Notes

1. For new-item receipts, the service creates inventory receipt records using the `productId`; item status is not evaluated because there is no TO item sequence.
2. For existing TO items, the status update is driven by the `statusId` sent in the request. `ITEM_COMPLETED` is the current way to support receive-and-close.
3. The current REST resource for TO receiving is `/receipts`. Older `/receive` and `/receiveAndClose` documentation should be treated as deprecated.
